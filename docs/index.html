Client-side encryption

Your browser encrypts every file with AES-256-GCM before it touches the server. Each file gets its own random key, wrapped with your master key, derived from your password via Argon2id. The server only ever stores ciphertext.

Streaming without decrypting server-side

Videos are remuxed to fragmented MP4 on upload — via ffmpeg in the CLI, or ffmpeg WASM in the browser (loaded once in the background). No re-encoding, just container restructuring. Each encrypted chunk is independently decodable. Your browser fetches 4 chunks in parallel, decrypts in a Web Worker, and appends to a MediaSource SourceBuffer. Playback starts after the first chunk.

Metadata is encrypted too

File names, MIME types, sizes, dimensions, durations — all encrypted into a single blob. The server stores it opaquely. Folder structures are also encrypted. Upload timestamps are coarsened to year + week to minimize leakage.

Chunk padding defeats fingerprinting

Every chunk is padded to a bucketed size (1, 2, 4, 8, or 16 MB) with random fill. An observer sees a handful of uniform sizes on disk — they can't determine original file sizes even with full disk access.

Secure deletion, not just unlinking

Deleted files are overwritten 3 times with random data, fsynced, then unlinked. On spinning disks this provides strong assurance. On SSDs it's best-effort due to wear leveling — an inherent hardware limitation.

Where most encrypted storage stops, Darkreel keeps going

Encrypting files is table stakes. What matters is everything around the files.

Get running on a VPS

One script, one binary, automatic HTTPS. No Docker, no Node.js, no external databases.

1

Get a server

Any Linux VPS. A $6/month DigitalOcean droplet or ~$4/month Hetzner VPS is more than enough. Point a domain at its IP.

2

Run setup

SSH in and run three commands. The script handles system updates, firewall, fail2ban, SSH hardening, Go, Caddy (automatic TLS), systemd service, and daily database backups.

root@vps ~
$ git clone https://github.com/baileywjohnson/darkreel.git $ cd darkreel $ sudo ./setup.sh [+] Domain: media.example.com [+] DNS check passed: media.example.com -> 137.184.86.254 [+] System updated [+] UFW firewall enabled (SSH, HTTP, HTTPS only) [+] fail2ban enabled [+] Caddy configured (automatic HTTPS via Let's Encrypt) [+] Daily database backup configured (3 AM, 7-day retention) Darkreel is running!
3

Save your recovery code

The script displays a 256-bit recovery code. Save it offline. It's the only way to recover your data if you forget your password. No one — including the server admin — can decrypt your files without it.

4

Upload

Open your domain, log in, and start uploading. Photos and videos are encrypted in your browser before they ever reach the server.

Hardened by default

The setup script doesn't just install Darkreel — it locks down the entire server.

+ Automatic TLS — Caddy + Let's Encrypt, zero config
+ Firewall — UFW: SSH, 80, 443 only
+ fail2ban — auto-bans brute force SSH
+ Rate limiting — 5 auth attempts/min per IP
+ Auto OS updates — unattended-upgrades
+ Process sandbox — NoNewPrivileges, ProtectSystem
+ SSH hardened — no root login, key-only auth
+ Session validation — logout is immediate
+ Daily backups — SQLite snapshots, 7-day retention
+ Auto-updates — optional, checksum-verified releases
+ SRI hashes — frontend tamper protection
+ Security headers — CSP, HSTS, X-Frame-Options

Also

+ Multi-user — isolated encrypted libraries, admin panel
+ Encrypted folders — only you see the structure
+ Recovery codes — 256-bit, shown once, your last resort
+ Single binary — Go + embedded frontend, no runtime deps
+ SQLite — no Postgres, no Redis, no external services
+ Upload to 50 GB — chunked, no size surprises
